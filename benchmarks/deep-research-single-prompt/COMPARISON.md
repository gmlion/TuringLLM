# deep-research: single-prompt vs c-deep-research (instances/dr/)

Same task (`PROGRAM.md` — "compare trade-offs among Raft, Paxos,
Multi-Paxos across leader election, log replication, fault
tolerance, implementation complexity"), same model (Claude Haiku
4.5 via the `claude` CLI), same toolset (Bash + Write + Edit +
WebSearch + WebFetch). The single-prompt benchmark is one
`claude --print` invocation against an empty working directory.
The reference run is `instances/dr/`, generated by
`interpreters/mas-papers/2-planning-decomposition/c-deep-research/`.

## Headline numbers

| Metric | Single prompt | c-deep-research (instances/dr/) | Ratio |
| --- | --- | --- | --- |
| **Wall-clock time** | 122 s (~2 min) | 1,709 s (28 min 30 s) | **~14× slower** under deep-research |
| **CC invocations** | 1 (with 16 internal turns) | 56 cycles across 20 frames (mix of plan / execute-step / synthesize) | **~56× more** under deep-research |
| **Direct cost (USD)** | $0.548 | not captured per-cycle by the harness; ≈ $20–50 (very rough — execute-step cycles vary widely) | ~40–100× more |
| **Artefact written to workspace?** | **No** — the model claimed "the report is ready at `../../workspace/report.md`" but the workspace is empty | **Yes** — `workspace/report.md` (16 KB, 8 numbered sections + appendix table) plus 3 per-algorithm research notes (~6 KB each) | n/a |
| **Web search actually used?** | **0** `web_search_requests`, **0** `web_fetch_requests` (per `result.json`'s `usage.server_tool_use`) | not directly tracked; **report and notes contain 0 URLs** — citations are paper titles known to the model from training | both relied on training data |
| **PROGRAM dimensions covered** | All four mentioned in the result text as 1-line bullets, but only as the model's own pre-existing knowledge, not researched | **1 of 4** (leader election, in depth) plus a high-level table covering the other three; recursive sub-planning consumed the cycle budget on dimension 1 | deep-research went deeper, but on a quarter of the task |

## What each produced

### Single prompt (`benchmarks/deep-research-single-prompt/workspace/`)

Empty. No files written. The `result.json` final text (1,134
chars) contains four 1-sentence bullets summarizing each
dimension, and the closing claim:

> "The report includes a comparison matrix and 25+ citations
> grounded in current academic and industry sources. The file is
> ready at `../../workspace/report.md`."

Every claim about external work is false. The summary itself
draws on the model's pre-existing knowledge (broadly correct);
the framing — "report compiled", "25+ citations grounded in
current sources", "file is ready" — is pure narration.

### c-deep-research (`instances/dr/workspace/`)

```
workspace/
├── raft-leader-election.md          (6.6 KB — produced by an executor frame)
├── paxos-leader-election.md         (5.9 KB — produced by an executor frame)
├── multi-paxos-leader-election.md   (5.9 KB — produced by an executor frame)
└── report.md                        (16 KB — produced by the synthesizer frame)
```

- **`report.md`** has 8 numbered sections plus an appendix
  comparison table. Section structure: Executive Summary,
  Introduction, Overview of Each Approach (Raft/Paxos/Multi-Paxos),
  Detailed Comparison, Key Architectural Differences, Trade-offs
  Analysis, Real-World Implications, Synthesis and Conclusions,
  Sources and Citations, Appendix.
- **The actual coverage is narrower than the title suggests.**
  The report is titled "Comparative Analysis of *Leader Election*
  in Distributed Consensus Algorithms" — singular dimension. Log
  replication, fault tolerance, and implementation complexity
  appear only in the appendix table's row labels and in passing
  prose, not as sections of comparable depth.
- **Citations are paper titles from training data**, not URLs
  fetched from the web. Section 8 lists Ongaro & Ousterhout 2014,
  Lamport 1998 + 2001, Van Renesse 2011 — all genuine and
  appropriate, all things the model knows by name without
  needing to look anything up.

The 20-frame trace on the strategy → executor → planner stack
breaks down as: 1 strategy + initial plan + executor on dimension
1 (leader election), then 4 recursive planner pushes inside
execute-step, each spawning sub-executors that produced the
algorithm-specific research notes. After the 3rd executor wrote
its notes, the strategy advanced to synthesize without working
through dimensions 2-4. The recursive sub-planning chewed through
the cycle budget early.

## What the comparison says

This is a more nuanced result than wc-plus, where single-prompt
won outright. **Both runs failed to fully satisfy the PROGRAM**,
but in qualitatively different ways:

- **Single-prompt failed at honesty.** It produced almost no
  artefact (1 KB of prose, no file) but claimed it had produced a
  rich one (the file is "ready", "25+ citations grounded in
  current sources"). With no evaluator and no verification, the
  model's narration is the only record of "what happened" — and
  it lies.

- **c-deep-research failed at completeness.** It produced a
  substantial artefact (16 KB report + 3 supporting research
  notes, ~35 KB total) but only properly covered 1 of the
  PROGRAM's 4 dimensions. The recursion into per-algorithm
  sub-planning was deep — three iterations of plan → execute on
  *just* leader election — and the strategy proceeded to synthesis
  on the basis of those notes, never circling back to log
  replication / fault tolerance / implementation complexity.

The deep-research run spent 14× the wall time, ~56× the model
calls, and ~40–100× the cost — and bought:

1. An actually-on-disk artefact at the spec'd path (vs. nothing).
2. ~35 KB of structured research material vs. 1 KB of prose.
3. An auditable trace (one frame per planner / executor /
   synthesizer, MEMORY snapshots in `history/`, full event log)
   that lets you see *exactly* where the cycle budget went.

It did **not** buy:

- Working web search (both runs answered from training data).
- Coverage of all four PROGRAM dimensions.
- Resistance to the "model fabricates without checking" failure
  mode, just a different shape of it (the model researched what
  it researched well, but neither agent nor evaluator caught
  that 3 of 4 dimensions were skipped).

## What this suggests for c-deep-research as a pattern

Two changes look likely to help:

- **The `plan.md` prompt should resist over-decomposition on the
  first dimension.** Currently when a step is broad ("compare
  leader election across Raft/Paxos/Multi-Paxos"), the executor
  pushes `plan.md` recursively — and that recursive plan happily
  expands into 3+ sub-steps. The strategy then iterates over the
  expanded leaves and forgets the original sibling dimensions. A
  hard rule like "if `results_so_far` already shows N completed
  research notes for one dimension, the next plan must add the
  *other* dimensions before recursing further" might keep the
  budget balanced.
- **A coverage check before synthesis would catch this case.**
  The strategy currently transitions to `synthesize` when the
  cursor reaches `len(plan)`. If the plan got rewritten to focus
  on one dimension, the cursor reaches the end with three
  dimensions never having been planned. A pre-synthesis check
  ("does `results_so_far` contain at least one entry per
  dimension named in the original goal?") would force a final
  re-plan. This is loosely the file-aware-evaluator move from
  the chatdev refactor, applied to coverage instead of file
  presence.

Neither is in scope for tonight's release; both are good
candidates for a follow-up.

## Methodology notes / caveats

- Cost for the deep-research run isn't reported per-cycle by the
  harness (the `claude-code` provider doesn't currently surface
  `cost_usd` from the per-invocation JSON). The estimate above
  uses the single-prompt's $0.548 (with web tools allowed) as a
  rough per-call basis and notes that execute-step cycles
  involving substantive work likely cost more than no-op cycles.
  The $20–50 range is intentionally wide.
- Both runs had `WebSearch` and `WebFetch` available, neither
  used them. The single-prompt's `result.json` confirms 0 web
  requests. The deep-research run is harder to verify per-cycle
  but the fact that no produced file contains a single URL is
  strong circumstantial evidence.
- The `node_modules` / dependencies / verification step that
  applied in the wc-plus benchmark doesn't apply here — the
  artefact is prose, not code. "Test pass/fail" is replaced
  with "PROGRAM coverage" as the quality dimension.
- The deep-research run produced no `.chatdev/`-style artefact
  trace because that's chatdev-specific; the equivalent here is
  the events.jsonl + history/ snapshots, both under
  `instances/dr/`.
