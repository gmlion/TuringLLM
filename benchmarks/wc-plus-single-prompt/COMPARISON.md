# wc-plus: single-prompt vs ChatDev (demo4b)

Same task (`PROGRAM.md`), same model (Claude Haiku 4.5 via the
`claude` CLI), same toolset (Bash + Write + Edit). The single-prompt
benchmark is one `claude --print` invocation against an empty
working directory. The ChatDev run is `instances/demo4b/`,
generated by `interpreters/5-fixed-sop-teams/b-chatdev/`.

## Headline numbers

| Metric | Single prompt | ChatDev (demo4b) | Ratio |
| --- | --- | --- | --- |
| **Wall-clock time** | 47 s | 1,444 s (24 min 4 s) | **~31× slower** under ChatDev |
| **CC invocations** | 1 (with 11 internal turns) | 43 cycles | **43× more** under ChatDev |
| **Direct cost (USD)** | $0.131 | not captured per-cycle by the harness; ≈ $0.10–0.20 × 43 ≈ **$4–9** | roughly **30–60× more** under ChatDev |
| **`npm test` result** | **11/11 pass** (verified post-hoc with `npm test`) | 9/9 pass per evaluator's inner check; not re-verified here | both green |

## What each produced

### Single prompt (`benchmarks/wc-plus-single-prompt/workspace/`)
```
src/wc-plus.js
src/wc-plus.test.js
package.json
README.md
```
- 11 tests including: standard format, JSON, help, version, missing-file
  error, whitespace handling, empty file. **All 11 pass.**
- `package.json` has the spec'd `bin.wc-plus` and `test` script.
- README ~857 bytes.

### ChatDev (`instances/demo4b/`)
```
workspace/
├── wc-plus.js
├── test
├── test_debug.js
├── README.md (1840 bytes)
└── .chatdev/
    ├── program.md
    ├── design.md       (1224 bytes)
    ├── code.md         (1493 bytes)
    ├── test_report.md  (754 bytes)
    ├── documentation.md (1265 bytes)
    └── last_rejection.md (975 bytes)
```
- The `.chatdev/` directory is overhead — design docs, test reports,
  rejection feedback. None of it is the artefact a user would
  consume; the artefact is `wc-plus.js` + `README.md`.
- 1 retry fired in the testing phase (the file-aware evaluator caught
  a missing `test_report.md` on the first attempt).

## What the comparison says

The ChatDev run is decisively more expensive on every dimension that
matters for a one-shot artefact like this — wall time, model calls,
dollar cost — and produces **roughly the same end-user output** (a
working CLI tool with tests and a README). For a task this size,
single-prompt dominates.

That said, the comparison isn't quite apples-to-apples for what
ChatDev is *trying* to do. Two things ChatDev produces that the
single prompt doesn't, and that might matter on different tasks:

1. **An auditable trace of the design rationale.** The `.chatdev/`
   directory contains the design doc, the consensus on what code was
   built, the test report, and the doc rationale — each as the
   negotiated agreement between two agents at that phase. For a
   one-shot CLI tool nobody cares; for a multi-week feature where
   you want to answer "why did we build it this way?", the trace
   may be worth something.
2. **A reviewer agent that can reject.** The single-prompt run is
   accepted purely on its author's say-so. ChatDev's evaluator
   actually inspected `workspace/` and rejected the first testing
   attempt for not producing a consensus artefact. On a harder task
   where the producer might silently skip a requirement, that
   independent verification could catch failures the single prompt
   wouldn't.

Whether either of those is worth a 30× cost multiplier depends on
the task. For wc-plus: clearly not. For something like a
multi-component refactor where requirements are easy to skip and
review evidence has long-term value: maybe.

## Methodology notes / caveats

- Cost for the ChatDev run isn't reported per-cycle by the harness
  (the `claude-code` provider doesn't currently surface `cost_usd`
  from the per-invocation JSON). The estimate above assumes each
  cycle's CC invocation costs roughly the same as the single-prompt
  one ($0.131 of which $0.13 is cache-creation). In practice
  per-cycle cost varies a lot: cycles where the model just runs
  awk to prune MEMORY are much cheaper than cycles where the
  coder writes wc-plus.js. The 30–60× range tries to bracket this.
- Both runs used Haiku 4.5 with cache. The single-prompt run shows
  460K cache-read tokens — that's Claude Code's default system
  prompt being read from cache rather than re-created. For ChatDev,
  every cycle pays cache-creation cost on its first call (because
  our shell passes `--system-prompt-file` per invocation), so the
  cache benefit is per-cycle, not per-run.
- The single-prompt run had a one-line preamble redirecting the
  PROGRAM's `../../workspace/` references to "current directory",
  since the benchmark has no nested frame layout. Functionally
  equivalent.
- 1 retry fired during demo4b — the testing phase's first
  evaluator-pass rejected for a missing `test_report.md` file.
  Without that retry, the cycle count would be ~37 instead of 43.
